# L2：AI 元数据评价设计

> 基于 API 元数据 + AI 模型，不依赖视频文件
> 填补 L1（纯统计）和 L3（视频分析）之间的关键断层
> 2026年5月

---

## 一、L2 的定位

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  L1（纯统计）          L2（AI+元数据）       L3（AI+视频） │
│  ───────────          ──────────────       ──────────── │
│                                                        │
│  回答"是多少"          回答"意味着什么"      回答"为什么好" │
│                                                        │
│  播放量 128K           这不是普通广告片，      00:04的Match Cut
│  收藏率 1.97%          它的视觉风格是          用光斑遮罩实现
│  团队 20人             "工业暗黑+碎切"，      时空跳跃，制造
│  标签: 梦幻/律动/风    对标的是潮玩和          魔幻现实主义基调
│                       游戏行业的顶级TVC       ...
│                                                        │
│  输入: 结构化数字      输入: 元数据+LLM推理    输入: 视频帧+LLM  │
│  方法: 统计算法        方法: LLM+ML+Embedding  方法: 多模态LLM  │
│  成本: ¥0            成本: ~¥0.05/条         成本: ~¥1.5/条   │
│                                                        │
└─────────────────────────────────────────────────────────┘
```

**L2 的核心价值**：在无法获取视频文件的情况下，用 AI 从元数据中"推理"出 L1 统计学无法给出的洞察。

### 模块2：ML 模式识别

#### 2.1 互动模式分类

```
不依赖LLM，纯ML分类模型（Random Forest / XGBoost）

输入特征：
  · count_view          播放量
  · count_like/view     点赞率
  · count_collect/view  收藏率
  · count_share/view    分享率
  · count_comment/view  评论率
  · collect/like        收藏/点赞比（>1 说明"参考意图"强）
  · like/view vs collect/view 比值
  · score/view          平台分/播放比
  · log(count_view)     对数播放量
  · log(count_follower) 对数粉丝数

分类标签（从训练数据中聚类后人工命名）：
  Type A："爆发型" —— 高播放+高分享率+中等收藏率
  Type B："参考型" —— 中等播放+极高收藏率+高收藏/点赞比
  Type C："口碑型" —— 低播放+极高点赞率+极高评论率
  Type D："长尾型" —— 低播放+中等互动率+持续缓慢增长
  Type E："普通型" —— 所有指标接近品类均值
  Type F："异常型" —— 指标组合偏离正常分布（疑似刷量）

华为Nova分类结果：
  → "参考型" (置信度 92%)
    依据：收藏/点赞比 1.94（>1阈值），收藏率1.97%（池内#1）
          播放量12.8万（非爆发型但也不低），分享率1.22%

价值：广告商搜"参考型"作品时优先展示。
```

#### 2.2 质量层级预测

```
目标：在 L1 质量评分 (quality 1-5) 基础上，用 ML 给出更细粒度的质量预测

训练数据：平台 quality 标签 + 社区信号 + 团队信号

模型输入：
  · quality             平台质量分
  · team_user_count     团队人数
  · team_role_diversity 角色覆盖度
  · director_popularity 导演人气
  · director_recommend_rate 导演推荐率
  · has_jinque          是否有金雀奖
  · has_jinshi          是否有金狮奖（从verify_desc提取）
  · log(count_view)     对数播放量
  · publish_dayofweek   周几发布
  · duration            时长

输出：0-10 连续质量分（比原始 1-5 更细腻）

华为Nova：
  原始 quality = 4 → 映射 8.0
  ML预测 = 9.1（团队+导演+金狮奖信号拉高了预测）
```

#### 2.3 异常检测

```
检测目标：刷量/刷赞/刷收藏行为

异常特征：
  · 收藏率异常高但评论率异常低 → 可能刷收藏
  · 播放量增速不符合平台自然增长曲线 → 可能刷播放
  · 点赞/评论/收藏比例严重偏离品类分布 → 可疑
  · 同一IP/设备短时间内大量互动 → 确认作弊
  · 播放量高但 score(平台分) 低 → 平台算法已识别为低质

方法：Isolation Forest / Elliptic Envelope（无监督异常检测）

输出：
  {
    "is_suspicious": false,
    "anomaly_score": 0.12,  // <0.3 → 正常
    "flagged_dimensions": []
  }

注意：异常检测结果不直接展示，仅用于内部降权或人工审核。
```

### 模块3：Embedding 语义理解

#### 3.1 作品语义向量生成

```
目标：将每部作品的元数据编码为一个稠密向量，用于相似度计算

编码内容：
  text = f"""
  标题：{title}
  分类：{category}
  标签：{tags}
  导演风格：{author.verify_desc}
  团队特征：{team_roles_summary}
  """

  方案A（轻量）：用 text embedding 模型（如 text2vec-large-chinese）
       → 1024维向量，本地推理，成本¥0
       → 适合大规模（百万级）作品索引

  方案B（精准）：用 LLM 生成语义描述再 embedding
       → 先让LLM把元数据总结为一段200字风格描述
       → 再对这段描述做 embedding
       → 成本~¥0.003/条，但语义更准确

建议：方案A做全量索引，方案B做Top 10万头部作品精细索引
```

#### 3.2 相似作品推荐

```
给定作品 → 向量相似度 Top 20 → 过滤同作者 → 过滤已看过 → Top 8 推荐

华为Nova (向量) ⟹ Top 8 相似：
  1. 小鹏汽车 MONA     余弦相似度 0.87  （同为新势力品牌态度片）
  2. HUAWEI Mate70     余弦相似度 0.84  （同为华为，不同产品线）
  3. 闲鱼「节后鱼生」    余弦相似度 0.79  （同为年轻化情绪碎切）
  ...

展示："相似风格"推荐卡片（附带相似度标签 + 关键相似点）
```

#### 3.3 风格聚类

```
全站10万+作品 → embedding → UMAP降维 → HDBSCAN聚类

可能出现的聚类（自动生成，人工命名）：
  · 聚类A："暗黑工业风·碎切·手机"          (华为Nova所在簇)
  · 聚类B："暖调自然光·慢节奏·生活品牌"
  · 聚类C："高饱和撞色·快节奏·美妆"
  · 聚类D："黑白极简·长镜头·奢侈品"
  ...

价值：
  · 自动发现平台的风格分布
  · 每种风格标注"代表作品Top 10"
  · 发现"风格空白"（市场需求大但平台上作品少的风格）
```

### 模块5：趋势与机会发现

#### 5.1 品类趋势检测

```
对每个品类，按周统计：

  · 新增作品数 → 品类热度
  · 平均播放量 → 品类流量
  · 平均收藏率 → 品类"参考价值"
  · 新入创作者数 → 品类吸引力
  · 品牌分布 → 品类商业化程度

统计分析：
  · 7日/30日/90日 移动平均 → 趋势方向
  · 同比/环比增长 → 拐点检测
  · 品类间交叉 → "广告商在投什么？"

输出示例：
  "手机数码类 2026Q1趋势：品牌态度片占比从35%上升到52%，
   功能片下降。品牌方从'讲参数'转向'讲态度'。"
```

#### 5.2 上升创作者识别


## 六、技术实现

### 6.1 模型选型

| 模块 | 模型 | 部署方式 | 单次成本 |
|------|------|---------|---------|
| LLM推理(1.1-1.4) | Qwen2.5-7B / DeepSeek-V3 | 本地GPU或云端API | ¥0.002-0.005 |
| ML分类(2.1-2.3) | XGBoost + Isolation Forest | CPU即可 | ¥0 |
| Embedding(3.1-3.3) | text2vec-large-chinese | 本地GPU | ¥0 |
| 综合推理(6.1) | DeepSeek-V3 / Qwen-Max | 云端API | ¥0.01 |

**汇总：单条作品 L2 总成本 ~¥0.05**

### 6.2 计算流程

```
作品发布/更新 → 消息队列 → L2分析Worker

Worker流程：
  1. 读取API元数据 (5ms)
  2. 并行执行模块1-5 (各模块独立，可并发):
     ├── LLM推理 (模块1.1-1.4)  ~2000ms (可合并为1次LLM调用)
     ├── ML分类 (模块2.1-2.3)    ~50ms
     ├── Embedding (模块3)       ~100ms  
     ├── 创作者分析 (模块4)      ~200ms (可缓存)
     └── 趋势分析 (模块5)         ~100ms (可缓存)
  3. 综合推理 (模块6)             ~500ms
  4. 写入结果缓存 + 搜索索引       ~50ms

总耗时：~3秒/条
并发处理：10 workers → 日处理28万条
```

### 6.3 缓存策略

```
缓存分级：
  L1 结果：T1缓存（1小时）
  模块1-2：T2缓存（24小时，因元数据变化慢）
  模块3：  T2缓存（7天，embedding不常变）
  模块4：  T3缓存（新作品发布时刷新）
  模块5：  T4缓存（每小时刷新品类统计）
  模块6：  T2缓存（依赖模块1-5）

触发重算：
  · 作品社区数据显著变化（播放量+5000 或 新增榜单）
  · 创作者发布新作品
  · 品类基准更新（24小时周期）
```

## 七、降级策略

```
当 LLM API 不可用时：
  → 模块1 回退到规则匹配（基于标签+分类的词典映射）
  → 模块6 回退到模板填充
  → L2 仍可产出基础版分析，但"洞察"深度下降

当 ML 模型不可用时：
  → 模块2 回退到阈值规则（收藏率>1.5%→参考型等）
  → 准确度从~90%降至~70%

当所有AI模块不可用时：
  → L2 整体降级为 L1+
  → 仅输出L1的统计评分 + 规则标签
  → 前端标注"AI分析暂时不可用，展示基础评分"
```
